SPAA: Secure and Private Agentic AI

A secure, private agentic AI interface. Your data and AI are sandboxed from the internet, and your traffic is either encrypted or off the public internet entirely.

Goals

Enables individuals, enterprises, friends, community groups etc, to have a private and secure AI environment that they share and build together. The interface and all data reside on a server, which is sandboxed from the internet. The AI can only see the data you have uploaded to your Knowledge Collections. The setup requires many steps, however once completed, you and your users can connect from most types of devices by simply installing one security app and opening a browser. Fully scalable setup - your AWS servers can be upgraded to accomodate more users and bigger LLMs when required.

Introduction

A lightweight, private and secure agentic AI console
A client-side (local) browser application using a 70 KB html file
The instance of Open WebUI on an AWS server requires only 2 vCPUs and 2GB RAM
Open WebUI and LLM(s) are each located on AWS servers, no local hardware required
Traffic between your device and AWS server with Open WebUI is via Tailscale (encrypted tunnel)
The AWS server hosting your Open WebUI has no public ports open
Traffic between your server and your private LLM on your Endpoint is encrypted or via AWS PrivateLink
Choose from thousands of different models on Hugging Face and host private LLMs
Once setup, other users can share your network - add users in Open WebUI and Tailscale
Use the LLM network and SPAA console across your enterprise - scales easily
Your LLM costs are managed in your Hugging Face account
The code is simple, 100% transparent, and is easy to adapt to your personal or professional needs

Setup

Below are setup instructions. Hugging Face provides access to free, open source LLMs. Once you have joined Hugging Face you can use the Inference Playground on their website and select a smart LLM to assist you set SPAA up (eg moonshotai/Kimi-K2-Instruct-0905).
Open WebUI is an open source, self-hosted AI platform that you setup on an AWS server, and which manages LLM deployment for you (and your users).
Tailscale is an identity-based connectivity platform that securely connects user devices with the AWS server.

Create an account with Hugging Face. HF has a free tier for using Inference. You can consider subscribing and/or buying credits to get more inference later. Go into your account settings and create an API Key.

Create an AWS account, then launch an EC2 instance:
STEP 1: LAUNCH EC2 INSTANCE

TIP - Setup Sections 11 and 12 note that seting up AWS Private Link is much easier if both your AWS servers are located in the same Region. If you will definitely use AWS Private Link you may want to read Setup Sections 11 and 12 before selecting your Region. Otherwise, there are simple workarounds you can do later, or just start a new AWS server later.

1. AWS Console > EC2 > Launch Instance
2. Settings:
- Name: open-webui
- AMI: Ubuntu 24.04 LTS
- Instance type: t3.small (2 vCPU, 2GB RAM, ARM-based, free tier eligible)
- Key pair: Create new one, download the .pem file, save it somewhere safe
- Storage: 8-30GB (default is fine to start)
3. Security group - set inbound rules:
- SSH (port 22) from "My IP" only
- That's it for now. No other ports needed.
4. Click Launch
NOTE: You may see a warning about "instance store not supported" - just ignore it and continue. It doesn't affect anything.
NOTE: AWS will warn about 0.0.0.0/0 rules if you add any. That's why we only add SSH from your IP.

STEP 2: SSH INTO YOUR INSTANCE
Mac/Linux:
chmod 400 /path/to/your-key.pem
ssh -i /path/to/your-key.pem ubuntu@<your-ec2-public-ip>
Windows (PowerShell):
ssh -i C:\path\to\your-key.pem ubuntu@<your-ec2-public-ip>

STEP 3: INSTALL DOCKER
Once SSH'd in:
sudo apt update && sudo apt upgrade -y
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER

COMMON ISSUES:
1. Windows SSH permissions
Error: "Permissions for .pem are too open"
Fix: icacls commands to restrict file permissions
2. Docker permission denied
Error: "permission denied...docker.sock"
Fix: Run "newgrp docker" or use "sudo docker"

RUN OPEN WEBUI
sudo docker run -d \
--name open-webui \
--restart unless-stopped \
-p 3000:8080 \
-v open-webui-data:/app/backend/data \
-e ENABLE_OLLAMA_API=false \
ghcr.io/open-webui/open-webui:main
Wait a minute or two for the container to pull and start.
Check it's running: sudo docker ps

OPEN AND CONNECT OPEN WEBUI
1. Go to AWS Console > EC2 > your instance > Security tab
2. Click the Security group link
3. Edit inbound rules > Add rule:
- Type: Custom TCP
- Port: 3000
- Source: My IP
4. Save
5. Open browser: http://<your-ec2-public-ip>:3000
6. Create your admin account (first user becomes admin)

CONNECT OPEN WEBUI TO HUGGING FACE
1. Click your profile icon (bottom left)
2. Go to Settings > Connections (or Admin Settings > Connections)
3. Under the OpenAI API section:
- URL: https://router.huggingface.co/v1
- API Key: Your HF token (from https://huggingface.co/settings/tokens)
4. Save
5. Go back to chat, click the model dropdown at the top
6. Select your model and start chatting to confirm it works

INSTALL TAILSCALE ON EC2
SSH into your EC2 instance:
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up
It will give you a URL to authenticate - open it in your browser and log into your Tailscale account. Your EC2 will appear in your Tailscale network.

INSTALL TAILSCALE ON YOUR DEVICES
- Laptop: Download from https://tailscale.com/download
- Phone: Install "Tailscale" from App Store / Play Store
- Sign in with the same Tailscale account on all devices
Open the Tailscale app on any device to see all connected machines and their Tailscale IPs (100.x.y.z format).

ACCESS OPEN WEBUI VIA TAILSCALE
1. Find your EC2's Tailscale IP:
- Open Tailscale app on your phone or laptop
- Look for your EC2 machine's IP (something like 100.x.y.z)
2. On your phone browser, go to:
http://100.x.y.z:3000
3. Log in with the same account you created in Step 5
4. Bookmark it or add to home screen for quick access
5. Same URL works from laptop too

LOCK DOWN SECURITY
Now that Tailscale is working, remove all public access:
1. Go to AWS Console > EC2 > your instance > Security tab
2. Click the Security group
3. Edit inbound rules
4. DELETE the port 3000 rule (no longer needed)
5. DELETE the SSH port 22 rule (SSH via Tailscale now)
6. Save - your security group should have NO inbound rules
(Tailscale works via outbound connections, needs no inbound rules)

VERIFY:
- Phone (on Tailscale): http://100.x.y.z:3000 - SHOULD work
- Laptop (on Tailscale): http://100.x.y.z:3000 - SHOULD work
- Public IP: http://<ec2-public-ip>:3000 - should NOT work (locked down)
- SSH now only via Tailscale: ssh -i key.pem ubuntu@100.x.y.z

OPTIONAL: RENAME EC2 IN TAILSCALE
In Tailscale admin console (https://login.tailscale.com/admin/machines), you can rename your EC2 machine to something like "openwebui-server". Then access it as: http://openwebui-server:3000

COMMON ISSUES:
Open WebUI not loading in browser
Error: "took too long to respond"
Fix: Port 3000 wasn't open in security group
Couldn't find HF connection settings
Wrong: Settings > Manage Tool Servers
Right: Settings > Connections > OpenAI API section

Personalise your configuration by updating the fields.

The system prompt gets injected into every chat. The model, tool, knowledge and skills are able to be defined seperately for each section, and will only be injected into chats within that section.
I deleted most of my personal fields but retained my default LLM model selections which match the basic functions of the agents for demonstration purposes of agentic setup, but these can be set to anything. Users can also make new LLM models by adding to base LLM models in Open WebUI, then call those models in config.js (see the Open WebUI docs).

You can add section tools, knowledge and skills. I kept some basic skill definitions here. You can do a great deal with strong skills definitions. The way Open WebUI handles knowledge is extremely efficient. You can set up multiple Knowledge Collections on different facets of your enterprise, then make them available to one or more Sections on your SPAA. Eg, materials, suppliers, contacts, invoices, contracts, training docs, product catalogues, reference materials, price lists, email contents, strategy docs, marketing campaigns, previous LLM chats and solutions, etc. Open WebUI will search Collection(s) via RAG for any relevant materials before it sends a query, then injects them into the chat. This allows you to selectively supply documents to your AI, instead of giving the AI open access to your data or systems.

Knowledge is managed in Open WebUI on the AWS server. You can read the Open WebUI help, but basically you create a Knowledge Collection and add files to it. It can be used as a Permanent Memory for your business and/or personal files, and an ongoing memory of your chats (for important chats that you want to refer to later (remember), download the chat file then upload it to your Knowledge Collection). Knowledge Collection(s) can be provided to a Section by adding the ID of the collection in config.js (click on the Knowledge Collection in Open WebUI and the ID is found in the web address - eg df6968fe-c557-2ecc-19ea-6f8723a1f7987). When a section has a Knowledge Collection assigned to it, Open WebUI does the retrieval from your knowledge collection (RAG) and injects the best-matching snippets into the prompt before it leaves.
TIP - upload a pdf of your product prices to Knowledge Collection then query an LLM about what is price of product X, to confirm you have it setup properly.

Privacy: The data in your Knowledge Collections are stored on your server in Open WebUI. Your privacy and security is maintained by AWS. You access the server via TailScale only, and closed your server public ports, therefore your server is not exposed to port attacks (Setup Section 3). Therefore your server and data traffic are protected.

No record is kept of the contents of any chats you have with your LLMs. However, some chats you will want to keep to refer to later. There are buttons to download your chats, and you can define where they are saved to in your browser. Upload these to a Knowledge Collection in Open WebUI. You can also compress the chat (which tells the LLM to summarise the chat prior to saving).

To use the AI console on a device, Tailscale must first be setup on that device, which you would have done on at least one device in Setup Section 3. Place the html and config.js files in the same folder on your device. Open the file directly in a browser.
TIP - You may get better performance (eg with the microphone) by starting a local server in a terminal (using for example Python on a PC), or using an App to start a server on mobile OS. You can also consider setting Tailscale to https instead of http to improve performance, however if you use a local server this is redundant.

Note that for security reasons the html file does not automatically go onto the internet to collect file converters for xlsx, docx and pdf files. You need to manually download them and place them in the same folder as the html file if you wish to do such conversions: pdf.worker.min.js, xlsx.full.min.js and mammoth.browser.min.js (google them and download).

Log into Open WebUI, go to Users tab - Invite User and enter their email. They will get a join link.
As Admin of the network, you can invite any users you wish. Users will install Tailscale and join your private network. They can access the Open WebUI network and can contribute to the Workspace, adding models, knowledge etc. Users can also modify their config.js to personalise their SPAA. Users do not need to do any of the above setup, that only needs to be done once by the Admin.

Most devices and OS have a Tailscale app (MacOS, iOS, Windows, Linux, Android). Install the app, the Tailscale Admin approves them joining the network, then the html file can be run. Each user can personalise their SPAA without modifying the html code via config.js.

You and your users can select from thousands of models on Hugging Face using Inference Providers (more than 10 Inference Providers whom Hugging Face contracts to host their models). Each LLM has a name (eg moonshotai/Kimi-K2-Instruct-0905) which is simply entered into the SPAA html page, and/or to config.js. These Inference Providers run the LLMs on their servers, and all have security and privacy policies. If you intend to run LLM(s) in the fully private and secure method you can try out different models first (eg without using any private data), then when ready you can deploy your favourite model(s) in private Inference Endpoint(s), as outlined below. Your SPAA Sections can refer to either an Inference Provider LLM or a private Inference Endpoint LLM.
Users can use their own Hugging Face accounts and pay for their own inference, or an enterprise might consider an enterprise plan with Hugging Face. Costs will be billed to the Hugging Face acocunt that owns the API key referenced in the Open WebUI Connection for the model that makes the query.

In Hugging Face go to Inference Endpoints. These work by Hugging Face creating an instance of an LLM on a server hosted by AWS (other providers are available). View the catalogue and select an LLM. Make this server private (in Authentication), such that the model is available 24/7 for your personal usage (and your invited users). You will get a private URL and then traffic between your server with Open WebUI and your server with your LLM will be direct with no third party (AWS server to AWS server). TLS/SSL encryption is used to secure data during transit. You can setup an even more secure transit method by enabling AWS Private Link (in Network) when you create your Endpoint. AWS sends traffic between your two AWS servers via private IP addresses on the AWS network, such that your traffic never traverses the public internet. This method can be useful for enterprise compliance purposes for sensitive data security.

There is a tutorial and docs on Hugging Face to demonstrate the setup. TIP - AWS has recently introduced setting up Private Link between servers in different regions, however its tricky and may need to be done manually (either copying your server image to a new server in another location or routing between locations). It is much easier if both your AWS servers are located in the same Region.

Your AWS server hosting your LLM is reached at a private URL. In Open WebUI go to Admin panel - Settings - Connections and create a new connection. Add the private URL (you may need to add /v1 at the end) and your HF API Key. You can leave the model name blank. To setup your private LLM its easier to first disable any other connections you have, so your not looking for it amongst 1000 others. Just open them and click disable (and return later and enable them again). Go Workspace - Models and add a new model. Select your private LLM model. Add a memorable name (no spaces) and save it (eg 'myendpoint'). Check it works - open a new chat and select 'myendpoint' and say hello. Back on your SPAA html page you can select the model using simply 'myendpoint', and you can update your config.js file to select this model by default.

You can have one or more sections in your SPAA using your private LLM, and other sections using the models from the Inference Providers on Hugging Face. The difference lies in how your traffic is handled. When you use your private LLM your traffic goes between two AWS servers, either encrypted or through Private Link. Using the Inference Providers your traffic goes from your AWS server with Open WebUI to one of more than 10 Inference Providers, whom Hugging Face contracts to host their models. All these Providers have privacy and security policies, but it is intrinsically less secure than using a private LLM.

If you used AWS for your server hosting Open WebUI, you may have gotten a free period initially, however server costs are quite low as Open WebUI has low computational demand. Hugging Face has different pricing options depending on your requirements. You start with a free account and you can setp up everything and start using models from the Inference Providers. You pay by token usage and you can find these prices on their webpage. Some of the small models can be used basically for free they are so cheap. However the latest and largest models cost a lot more than the small and less capable models. If you purchase a subscription you can access many more tokens per month, and you can also purchase credits.

Private Inference Endpoints are priced based on which model you host. The costs are directly related to the model size since you need to purchase enough RAM to run the model. You pay for server up-time, and can use as many tokens as you want while its up, and you are billed from your purchased credits. The server stops after 1 hour of inactivity. Note they are easy to start and delete, so you can try different models and if you dont like them just delete the Endpoint. You can also have many Endpoints and link them all to your AWS Private Link.
